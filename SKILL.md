# Batch Article Summary Skill

当用户要你总结一份很大的 JSON 文章集合，并且要求最终结果是可直接打开的网页时，使用这个 Skill。

## 触发场景

以下表达默认都应优先联想到这个 Skill：

- `批量总结`
- `批量总结文章`
- `批量总结这些内容`
- `批量总结这个 JSON`
- `总结多篇文章`
- `把这些文章整理成网页`
- `把 JSON 文章生成网页总结`
- `多篇文章汇总成 HTML`

如果用户给的是一个包含多篇文章的 JSON、对象数组或文章集合，即使没有明确说出 Skill 名，只要需求明显是“批量总结 + 网页输出”，也应优先使用这个 Skill。

## 输入约定

- 输入通常是一个 JSON 数组。
- 也可能是形如 `{ "articles": [...] }` 的对象包装结构；如果如此，默认取其中的 `articles` 数组继续处理。
- 每个元素至少包含 `title`、`url`、`content`。
- 必须按原始顺序处理，编号从 1 开始。

## 执行流程

1. 先确认输入是否是文章数组；如果不是，先做最小必要的结构修正。
2. 如果 JSON 很大或文章很多，不要一次把所有正文都塞进一次总结里。
   - 先分批阅读和整理，每批保留文章原顺序。
   - 每篇文章单独产出一个总结块，最后再汇总成一个完整 HTML。
3. 生成输出路径：
   - 运行 `bash scripts/next_output_path.sh`
   - 脚本会返回当前工作目录下可直接写入的 HTML 路径，固定放到 `outputs/batch-article-summary-skill/<yyyymmdd-内容名>/report.html`。
4. 按 [references/prompt-template.md](references/prompt-template.md) 的要求生成 **完整 HTML**，不要输出 Markdown。
5. 优先以 [assets/report-template.html](assets/report-template.html) 为页面骨架，替换标题、统计区、导航、文章卡片列表和页脚。
6. 将 HTML 保存到第 3 步返回的路径。
7. 保存后执行 `open <html-path>`，在默认浏览器中打开。

## 输出要求

- 必须是完整可独立打开的 HTML 文档，包含 `<!DOCTYPE html>`、`<head>`、`<body>`。
- 页面语言使用 `zh-CN`。
- 页面需强调易读性：
  - 顶部要有总标题和文章数量。
  - 每篇文章使用独立卡片，标题可点击跳转原文。
  - 软文和无法总结的内容要用明显的提示样式区分。
- 不要把原始大段正文直接塞进页面。
- 忠于原文，不补编不存在的信息。
- 普通文章的默认结构应与提示词保持一致：
  - 推荐指数
  - 这篇最值得知道的内容
  - 关键细节
  - 深度解读
  - 最终判断
- 其中最核心的是“这篇最值得知道的内容”：
  - 不是机械复述原文
  - 也不是固定死条数的流水账
  - 通常控制在 3 到 5 条，但要根据文章信息密度灵活判断，宁少勿滥
  - 每条都先讲清重点，再带一个最值得记住的钩子
- “关键细节” 只补 1 到 3 个值得单独拎出来的具体信息，不要把前面的重点再完整讲一遍。
- “深度解读” 需要加入适度、克制、基于原文的批判性判断：
  - 重点看论证是否充分、前提是否站得住、哪些是事实、哪些只是作者判断
  - 不要为了显得聪明而乱挑刺
- “最终判断” 要短、干脆，像一句收束性的编辑判断。
- 不要把总结写成一整块长文。
- 内容分段要明显，尽量一句一段，重点单独站出来。
- 如果整批文章之间没有明显内在联系，不要在页头强行写“它们共同在讨论什么”。
- 此时页头只需要说明：
  - 这是一次批量总结/阅读列表整理
  - 文章按原顺序排列
  - 可用于快速回看和筛选是否值得细读

## 大体量 JSON 的处理原则

- 优先保证“每篇都覆盖到”，其次再追求文采。
- 当单篇正文特别长时，先提取核心观点、案例、数字和结论，再写总结块。
- 如果中途需要暂存，可先在内存里维护“编号 -> 总结块”的映射，最后一次性组装 HTML。
- 如果某篇内容明显是广告、课程销售、产品推广，且推广是全文主目的、正文本身缺少独立信息价值，直接输出软文标记，不再强行总结。
- 如果文章主体分析是扎实的，只是在结尾带了一小段自家平台、产品、课程、公众号、链接或行动引导，不要直接判软文，仍按普通文章总结。
- 如果是活动体验、参会随笔、观展记录或线下见闻，只要正文有具体见闻、人物观察、现场细节或独立观点，即使存在主办方、品牌方、合作方露出，也优先按普通文章总结。
- 如果 `content` 为空、乱码、极短或信息不足，直接输出无法总结标记。

## 生成时不要遗漏

- 标题格式改成 HTML 链接，不再使用 Markdown 链接。
- 每篇都保留编号标识，例如 `article-1`、`article-2`，方便定位和后续扩展。
- 推荐指数统一用星标文本呈现，例如 `⭐⭐⭐⭐☆`。
- “这篇最值得知道的内容” 优先使用有序列表。
- 每条内容都要兼顾两件事：
  - 这条到底在说什么
  - 这条最值得记住什么
- 条数不要机械固定；能用 3 条讲明白，就不要硬凑 5 条。
- 每条内容优先写成“有重点 + 有钩子”的短信息块，而不是“提炼后的空结论”。
- 关键细节只保留真正值得展开一点点的具体东西，避免二次复述。
- 深度解读要有判断力，但要建立在理解原文的基础上。
- 最终判断必须干脆，别绕成另一段总结。
- 排版上优先做出“编号 + 重点句 + 记忆钩子 + 简短判断”的节奏，让人快速扫读也能带走重点。
- 页面阅读节奏优先靠分段和留白来建立，不要靠堆很多字来显得内容丰富。
